# **기술 요구사항 정의서 (TRD): Sell-Master 시스템 아키텍처**

## **1\. 시스템 아키텍처 개요**

본 시스템은 대량의 API 요청과 실시간 데이터 동기화를 처리하기 위해 확장 가능한 마이크로서비스 아키텍처(MSA)를 지향합니다.

## **2\. 기술 스택 (Tech Stack)**

* **Frontend:** Next.js (App Router), Tailwind CSS, React Query, Zustand.  
* **Backend:** Node.js (NestJS) 또는 Python (FastAPI).  
* **Database:** PostgreSQL (주 데이터), Redis (캐싱 및 작업 큐).  
* **Worker:** BullMQ 또는 Celery (대량 주문 수집 및 상품 동기화 비동기 처리).  
* **Infrastructure:** AWS (ECS, RDS, S3, CloudFront).  
* **Monitoring:** Sentry, Datadog (또는 Prometheus/Grafana).

## **3\. 핵심 기술 설계**

### **3.1. 외부 채널 연동 엔진 (Integration Engine)**

* **Adapter Pattern 적용:** 마켓별 API 규격이 다르므로 이를 추상화한 MarketAdapter 인터페이스 구현.  
* **Rate Limiting:** 마켓별 API 호출 제한(Rate Limit)을 준수하기 위한 Token Bucket 알고리즘 적용.  
* **Scraping (Fallback):** API가 제공되지 않는 기능의 경우 Puppeteer/Playwright 기반의 헤드리스 브라우저 활용.

### **3.2. 데이터 동기화 전략 (Sync Strategy)**

* **Webhook 처리:** 마켓에서 제공하는 웹훅(예: 네이버 스마트스토어)을 최우선으로 처리.  
* **Polling Worker:** 웹훅이 없는 채널은 주기적으로(예: 5\~10분 간격) 백그라운드 워커가 데이터를 수집.  
* **Optimistic UI:** 사용자 조작 시 프론트엔드에서 즉각 반영 후 백그라운드에서 성공 여부 확인.

### **3.3. 재고 정합성 관리**

* **Distributed Lock:** 재고 차감 시 레이스 컨디션을 방지하기 위해 Redis를 이용한 분산 락 구현.  
* **Eventual Consistency:** 대량의 상품 정보 수정 시 메시지 큐를 사용하여 순차적으로 마켓에 반영.

## **4\. 데이터베이스 스키마 주요 엔티티**

* **Users/Teams:** 사용자 계정 및 권한 관리.  
* **Stores:** 각 마켓별 API 키, 접근 토큰, 상태 정보.  
* **Products:** 통합 상품 정보 (Master Data).  
* **MarketProducts:** 각 마켓에 등록된 상품 상세 (마켓별 ID, 가격, 매핑 상태).  
* **Orders:** 통합 주문 내역 및 배송 상태.  
* **Inventory Logs:** 재고 변동 히스토리.

## **5\. 인프라 및 보안**

* **OAuth 2.0:** 마켓 연동 시 안전한 인증 절차 준수.  
* **Encryption:** 마켓 API 키 등 민감 데이터는 AES-256 암호화하여 DB 저장.  
* **CI/CD:** GitHub Actions를 통한 자동 빌드 및 배포 환경 구축.

## **6\. 확장성 고려 사항**

* **Auto-scaling:** 주문 수집이 몰리는 특정 시간대(예: 아침 출근 직후)에 맞춘 ECS 클러스터 자동 확장.  
* **Multi-region:** 추후 글로벌 진출 시 지역별 마켓 연동을 위한 인프라 분리 고려.
